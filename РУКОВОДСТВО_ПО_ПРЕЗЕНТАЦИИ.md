## Слайд 2 — Проблема и Решение

### Левая колонка — Проблема

Расскажи о текущем состоянии дел в общежитии. Подчеркни, что ты лично знаком с ситуацией как студент АВПК.

**Что сказать:**
> «Начну с проблемы. В нашем общежитии учёт ведётся в бумажном виде или в Excel-таблицах. Комендант вручную записывает, кто в какой комнате живёт, кто оплатил, кто убыл. Заявки на заселение подаются устно или через мессенджеры — нет единой базы.»

Раскрой каждую проблему подробно:

| Проблема | Как раскрыть |
|----------|-------------|
| **Бумажный учёт** | «Списки студентов хранятся в тетради или разрозненных файлах. При заселении нового студента комендант листает журнал, ищет свободное место — это долго и неудобно.» |
| **Нет онлайн-регистрации** | «Абитуриент не может сам посмотреть свободные места и подать заявку. Он вынужден лично прийти в общежитие или звонить коменданту.» |
| **Контроль оплат** | «Комендант должен помнить, кто и за какой месяц оплатил. Легко пропустить должника, особенно когда студентов больше 100.» |
| **Отсутствие истории** | «Нет возможности посмотреть, кто раньше жил в конкретной комнате или на конкретном месте. Информация теряется при выселении.» |

### Правая колонка — Решение

**Что сказать:**
> «Разработанная система решает все эти проблемы. Вместо бумажного журнала — электронная база данных. Вместо устных заявок — форма бронирования на сайте. Вместо памяти коменданта — автоматический учёт оплат и журнал убытий.»

## Слайд 4 — Стек технологий

Не просто перечисляй — объясняй **почему** выбрана каждая технология.

**Что сказать:**

| Технология | Почему выбрана |
|------------|---------------|
| **React + Vite** | «React — самая популярная библиотека для создания интерфейсов. Vite обеспечивает мгновенную сборку при разработке.» |
| **Express** | «Лёгкий и гибкий веб-фреймворк для Node.js. Идеален для REST API средней сложности.» |
| **Prisma ORM** | «Prisma генерирует типизированный клиент для работы с базой данных. Никакого сырого SQL в коде, полная автозамена в IDE, защита от SQL-инъекций на уровне библиотеки.» |
| **PostgreSQL** | «Надёжная реляционная база данных. Поддерживает транзакции, внешние ключи, уникальные ограничения — всё, что нужно для согласованности данных общежития.» |
| **JWT + bcrypt** | «JWT позволяет не хранить сессию на сервере. Токен подписывается криптографически, срок действия настраивается. Пароли хешируются bcrypt — даже если кто-то получит доступ к базе, он не увидит пароли админов.» |
| **CSS Modules** | «Стили изолированы на уровне компонента. Не нужно бояться конфликта имён классов. Никакого Tailwind и inline-стилей.» |
| **SVG + Figma** | «План этажа — векторная графика. Координаты каждой комнаты взяты из реального плана здания через Figma. При масштабировании не теряется чёткость.» |

**Как работает защита:**
> «При подаче заявки система ищет похожих студентов в базе. Телефон нормализуется — из номера удаляются все нецифровые символы, берутся последние 10 цифр. +7 (777) 123-45-67 превращается в 7771234567. 87771234567 превращается в то же самое. Имена сравниваются через расстояние Левенштейна — алгоритм подсчитывает, сколько символов нужно изменить чтобы превратить одно имя в другое. Если сходство больше 80 процентов и телефоны совпадают — заявка помечается флагом.»

**Почему не блокировка:**
> «Важный момент: система не блокирует такие заявки. Она лишь предупреждает. Потому что могут быть реальные случаи — два брата с похожими именами, или студент потерял доступ к прошлой заявке и подаёт новую. Решение всегда принимает человек — администратор.»

**Подтверждение заявки:**
> «При подтверждении заявки система в одной транзакции создаёт студента, назначает его в комнату на выбранное место, и меняет статус брони на "подтверждена". Если за те секунды, пока админ смотрел заявку, кто-то успел занять это место — транзакция откатится и админ увидит ошибку.»

# Ответы на вероятные вопросы

### Q: Сколько времени заняла разработка?
**A:** «Основная разработка велась в течение семестра. Было несколько итераций: сначала базовая структура и БД, потом CMS, потом публичный сайт, потом доработка UX и защиты от дубликатов.»

### Q: Как происходит развёртывание на сервере?
**A:** «Бэкенд запускается через `node index.js`, база данных через `prisma migrate deploy`, фронтенд и CMS собираются в статические файлы через `vite build` и раздаются через Nginx. В дипломном варианте всё работает локально, но для продакшена достаточно скопировать файлы на сервер и настроить process manager вроде PM2.»

### Q: Что если общежитие расширится — добавится 6-й этаж?
**A:** «Достаточно добавить комнаты в базу данных — через CMS или прямым запросом. Код не привязан к количеству этажей. Фильтры и план этажа подхватят новые комнаты автоматически.»

### Q: Насколько система защищена?
**A:** «Несколько уровней: JWT-авторизация на все админские эндпоинты, bcrypt для паролей, Prisma защищает от SQL-инъекций, проверка занятости мест в транзакции, нормализация телефонов и fuzzy-поиск для защиты от дубликатов заявок.»

### Q: Можно ли добавить оплату через Kaspi QR?
**A:** «Архитектурно — да. Модуль оплаты независим от способа платежа. Можно добавить интеграцию с платёжным шлюзом и при подтверждении платежа автоматически вызывать существующий API отметки оплаты.»

### Q: Как работает миграция базы данных?
**A:** «Через Prisma Migrate. Каждое изменение схемы создаёт SQL-файл миграции, который можно применить к любой копии базы данных. Миграции хранятся в репозитории и версионируются git-ом. Откат изменений делается одной командой.»

| **CRUD** | Create, Read, Update, Delete — базовые операции |
| **REST API** | Архитектура взаимодействия фронтенда с сервером |
| **ORM** | Object-Relational Mapping — Prisma |
| **JWT** | JSON Web Token — способ авторизации |
| **Миграция** | Версионирование структуры базы данных |
| **CSS Modules** | Изоляция стилей компонентов |

### Чего избегать

- **Не говори** «Тут всё просто» — комиссия может подумать, что проект тривиальный
- **Не извиняйся** за недоработки — лучше скажи «в планах по развитию»
- **Не читай со слайдов** — слайды для комиссии, ты рассказываешь своими словами
- **Не спеши** — 12 минут на 11 слайдов это нормально, делай паузы
- **Не молчи** во время загрузки страниц — комментируй что происходит
