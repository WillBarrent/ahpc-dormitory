# Заключение о проделанной работе

## Введение

Дипломный проект «Информационная система общежития Актюбинского высшего политехнического колледжа» представляет собой полноценное веб-приложение, автоматизирующее полный цикл управления студенческим общежитием — от подачи заявки на заселение до выселения студента. Разработанная система заменяет бумажные журналы и разрозненные Excel-таблицы единой цифровой платформой с разграничением ролей: публичный сайт для студентов и административная панель для коменданта.

---

## 1. Архитектура системы

Система построена по трёхкомпонентной архитектуре, где каждое приложение выполняет свою чётко определённую роль:

### 1.1. Публичный фронтенд (порт :5174)
React-приложение, доступное студентам без авторизации. Предоставляет:
- **Главную страницу** с информацией об общежитии, правилах проживания и контактными данными
- **Интерактивный план этажа** (SVG-графика) — визуальное представление всех пяти этажей с цветовой индикацией занятости комнат
- **Систему бронирования** — пошаговый процесс: выбор этажа → выбор комнаты на плане → выбор конкретного спального места → заполнение анкеты → подтверждение заявки
- **Страницу проверки статуса заявки** — студент вводит ФИО и телефон, чтобы узнать решение администратора по своей заявке

### 1.2. Административная панель CMS (порт :5173)
React-приложение с JWT-аутентификацией для коменданта. Включает:
- **Дашборд** — сводная статистика: количество заселённых студентов, свободных мест, оплативших за текущий месяц, необработанных заявок
- **Управление студентами** — полный список с поиском, сортировкой, пагинацией; раскрывающиеся строки с детальной информацией; добавление/редактирование/выселение; импорт из Excel; печать заявления на заселение
- **Управление комнатами** — сетка комнат по этажам с индикацией занятости мест; редактирование типа комнаты и количества мест
- **Обработка заявок (бронирований)** — список заявок с фильтрацией по статусу; подтверждение (с автоматическим созданием студента в транзакции) или отклонение
- **Учёт оплат** — ежемесячная отметка оплативших студентов с фильтрацией по месяцу, году и этажу; экспорт в форматированный Excel
- **Журнал убытий (отлучек)** — создание, подтверждение и завершение убытий студентов; печать расписки; отслеживание просроченных возвращений

### 1.3. API-сервер (порт :3001)
Express.js сервер, обслуживающий оба клиентских приложения. Реализует:
- **8 групп маршрутов**: `/api/auth`, `/api/students`, `/api/rooms`, `/api/bookings`, `/api/payments`, `/api/absences`, `/api/stats`, `/api/import`
- **Middleware авторизации** — проверка JWT-токена для защищённых маршрутов; публичные маршруты (бронирование, проверка статуса) доступны без токена
- **CORS-защиту** — запросы принимаются только с доверенных источников (:5173, :5174)
- **Prisma ORM** — типизированный слой доступа к базе данных, исключающий SQL-инъекции

### 1.4. База данных
PostgreSQL под управлением Prisma ORM. Спроектирована нормализованная схема из **7 таблиц**:

| Таблица | Назначение | Ключевые поля |
|---------|-----------|---------------|
| `Admin` | Учётные записи комендантов | login, password (bcrypt) |
| `Room` | Комнаты общежития | number, floor, totalBeds, type (RESIDENTIAL/NON_RESIDENTIAL/TOILET/TECH) |
| `Student` | Заселённые студенты | fullName, course, group, phone, roomId, bedNumber, status (PENDING/ACTIVE) |
| `Booking` | Заявки на заселение | fullName, course, group, phone, roomId, bedNumber, status (PENDING/CONFIRMED/REJECTED) |
| `Payment` | Ежемесячные оплаты | studentId, month, year, amount (уникальная пара student+месяц+год) |
| `Absence` | Отлучки студентов | studentId, startDate, endDate, reason, status (PENDING/ACTIVE/COMPLETED) |

Связи между таблицами: `Room → Student` (одна комната — много студентов), `Room → Booking` (одна комната — много заявок), `Student → Payment` (один студент — много платежей), `Student → Absence` (один студент — много отлучек), `Booking → Student` (связь для пометки дубликатов).

## 2. Функциональные возможности

### 2.1. Процесс бронирования (полный цикл)

Разработан сквозной процесс заселения студента:

1. **Студент** заходит на публичный сайт → выбирает этаж → видит интерактивный SVG-план → нажимает на комнату → видит сетку спальных мест с отметками «свободно/занято» → выбирает свободное место → заполняет анкету (ФИО, курс, группа, телефон)
2. **Система** проверяет: существование комнаты, её тип (только RESIDENTIAL), доступность места (не занято активным студентом и нет pending-заявки), допустимость номера койки
3. **Fuzzy-поиск дубликатов**: нормализация телефона (удаление нецифровых символов, приведение к единому формату), сравнение ФИО через расстояние Левенштейна (порог ≥80%). При обнаружении похожего студента заявка помечается флагом, но не блокируется — решение остаётся за администратором
4. **Администратор** видит заявку в CMS, анализирует флаг дубликата (если есть), принимает решение: **Подтвердить** (в транзакции создаётся Student + обновляется Booking → CONFIRMED) или **Отклонить** (Booking → REJECTED)
5. **Студент** проверяет статус заявки через страницу проверки на публичном сайте

### 2.2. Управление контингентом

- **Раскрывающиеся строки** — клик по студенту в таблице раскрывает полную информацию без перехода на отдельную страницу: контакты, комната, место, дата заселения, статус оплаты за текущий месяц, текущее убытие
- **Поиск с debounce** — поиск по ФИО с задержкой 400 мс, исключающей избыточные запросы к серверу при быстром наборе
- **Сортировка по столбцам** — кликабельные заголовки: ФИО, курс, группа, комната, этаж
- **Пагинация** — настраиваемое количество записей на странице (25/50/100)
- **Импорт из Excel** — пакетная загрузка студентов из .xlsx файлов
- **Печать заявления** — генерация готового юридического документа (заявление о заселении) через react-to-print

### 2.3. Финансовый учёт

- **Ежемесячная оплата** — отметка студентов, оплативших проживание за выбранный месяц
- **Статистика в шапке** — мгновенная сводка: сколько оплатили, сколько нет
- **Сумма оплаты с сервера** — не захардкожена в коде, изменение стоимости не требует пересборки приложения
- **Фильтры**: месяц, год, этаж
- **Защита от двойной оплаты** — уникальный constraint `[studentId, month, year]` на уровне базы данных
- **Экспорт в Excel** — форматированная выгрузка со стилями и автофильтром через xlsx-js-style

### 2.4. Журнал убытий

- **Отлучки студентов** — создание записи с датами начала/окончания и причиной
- **Машина состояний**: PENDING → ACTIVE → COMPLETED
- **Визуальные индикаторы**: просроченные возвращения выделяются пульсирующей анимацией
- **Печать расписки** — генерация документа об убытии
- **Интеграция с профилем студента** — текущее убытие отображается в раскрытой строке студента

### 2.5. Дашборд администратора

- **Ключевые метрики** в реальном времени: всего студентов, активных, ожидающих подписания, свободных мест, оплативших/неоплативших, новых заявок
- **Карточки-ссылки** — клик по метрике ведёт на соответствующую страницу управления
- **Визуальная индикация** — карточка с необработанными заявками подсвечивается рамкой, привлекая внимание администратора

## 3. Техническая реализация

### 3.1. Стек технологий

| Слой | Технология | Обоснование |
|------|-----------|-------------|
| **Публичный фронтенд** | React 18 + Vite, CSS Modules, React Router DOM v7 | Быстрая сборка, изоляция стилей, маршрутизация |
| **Админ-панель** | React 18 + Vite, CSS Modules, React Router DOM v7 | Единый стек с фронтендом, переиспользование подходов |
| **API-сервер** | Node.js + Express.js | Единый язык (JavaScript) на всём стеке |
| **ORM** | Prisma | Типобезопасные запросы, миграции, защита от SQL-инъекций |
| **База данных** | PostgreSQL | Надёжная реляционная СУБД, поддержка транзакций |
| **Аутентификация** | JWT (JSON Web Token) + bcrypt | Безопасное хеширование паролей, stateless-авторизация |
| **SVG-графика** | Plain SVG + React state | Нативные CSS-классы на SVG-элементах, никаких сторонних библиотек |
| **Генерация документов** | react-to-print | Печать заявлений и расписок прямо из браузера |
| **Excel** | xlsx-js-style | Чтение (импорт студентов) и запись (экспорт отчётов) с форматированием |

### 3.2. Количественные показатели

| Показатель | Значение |
|-----------|----------|
| **Строк кода (бэкенд)** | ~1 700 (включая маршруты, middleware, схему Prisma) |
| **Строк кода (публичный фронтенд)** | ~4 500 |
| **Строк кода (админ-панель)** | ~5 500 |
| **Общий объём кодовой базы** | ~11 700 строк |
| **Файлов проекта** | 178 (без учёта node_modules) |
| **Таблиц базы данных** | 7 |
| **API-маршрутов** | 8 групп (auth, students, rooms, bookings, payments, absences, stats, import) |
| **Страниц публичного сайта** | 3 (главная, план этажа, статус заявки) |
| **Страниц админ-панели** | 6 (дашборд, студенты, комнаты, бронирования, оплата, убытия) |
| **Интерактивных SVG-планов этажей** | 5 (по одному на каждый этаж) |

### 3.3. Обработка граничных случаев

- **Race condition при подтверждении заявки**: два администратора могут одновременно подтвердить одну заявку. Проблема решена через `$transaction` Prisma — перед созданием студента повторно проверяется доступность места
- **Защита от двойного бронирования**: место не может быть одновременно занято активным студентом и иметь pending-заявку — проверка на обоих уровнях (публичный API и транзакция подтверждения)
- **Fuzzy-дубликаты**: опечатки в ФИО (расстояние Левенштейна, порог 80%) и различия в формате телефона (нормализация) не позволяют обойти систему повторными заявками
- **Валидация на уровне БД**: уникальные constraint'ы (`studentId + month + year` для платежей, `number` для комнат) предотвращают логические ошибки даже при сбое приложения
- **Каскадное выселение**: при выселении студента (`movedOut`) место освобождается автоматически

## 4. Пользовательский интерфейс и опыт (UI/UX)

### 4.1. Единая дизайн-система

Все страницы админ-панели следуют единому визуальному стандарту:
- **Шапка страницы**: заголовок + кнопка действия + строка статистики (где применимо)
- **Панель фильтров**: поиск, выпадающие списки, переключатели статуса
- **Таблица**: фиксированные колонки с действиями, цветовые бейджи статусов, раскрывающиеся строки
- **Пагинация**: единый компонент с выбором количества записей на странице

### 4.2. UX-улучшения

- **Debounce-поиск** (400 мс) — мгновенный отклик без лишней нагрузки на сервер
- **Кастомные модальные окна** — подтверждение опасных действий (удаление, выселение) через стилизованные диалоги вместо браузерного `confirm()`
- **Закрытие по Escape** — все модальные окна закрываются клавишей Escape
- **Цветовое кодирование статусов**: зелёный (активен/оплачено/подтверждено), жёлтый (ожидает/не оплачено), красный (отклонено/просрочено)
- **Пульсирующая анимация** для просроченных убытий — мгновенно привлекает внимание
- **Интерактивный SVG-план**: наведение на комнату подсвечивает её и показывает всплывающую информацию о занятости (React `useState`, не CSS-трюки)

### 4.3. Адаптивность

Интерфейс админ-панели оптимизирован для использования на десктопе (ноутбук коменданта). Публичный сайт доступен с любых устройств — студенты могут подать заявку с телефона.

## 5. Безопасность

| Аспект | Реализация |
|--------|------------|
| **Аутентификация** | JWT-токены с ограниченным сроком жизни; пароли хранятся в bcrypt-хеше |
| **Авторизация** | Middleware проверяет токен на каждом защищённом маршруте; публичные маршруты (бронирование, статус) доступны без авторизации |
| **CORS** | Запросы принимаются только с доверенных источников (:5173, :5174) |
| **SQL-инъекции** | Исключены использованием Prisma ORM с параметризованными запросами |
| **Валидация ввода** | Все поля проверяются на уровне API перед записью в БД |
| **Транзакции** | Критические операции (подтверждение заявки → создание студента) выполняются атомарно |
| **Защита от дубликатов** | Двухуровневая: точное совпадение (занятость места) + нечёткое (fuzzy-поиск по имени и телефону) |

## 7. Выводы

Разработанная информационная система общежития АВПК обеспечивает:

1. **Полную автоматизацию** цикла управления общежитием: бронирование → подтверждение → заселение → оплата → учёт убытий → выселение
2. **Прозрачность процессов** — дашборд даёт мгновенную картину состояния общежития: кто заселён, кто оплатил, какие заявки ждут решения
3. **Защиту от ошибок и злоупотреблений** — двухуровневая проверка дубликатов, транзакции для критических операций, constraint'ы на уровне базы данных
4. **Удобство работы** — единая дизайн-система, поиск с debounce, сортировка, пагинация, кастомные модальные окна, горячие клавиши
5. **Масштабируемость** — система спроектирована без хардкода: справочники групп и специальностей, стоимость проживания, типы комнат — всё настраивается через интерфейс администратора без изменения кода
6. **Современный технологический стек** — React + Vite на фронтенде, Express + Prisma + PostgreSQL на бэкенде, единый язык программирования (JavaScript) на всех слоях

Система готова к внедрению в реальную эксплуатацию и может служить основой для дальнейшего развития: добавления личного кабинета студента, интеграции с бухгалтерскими системами колледжа, внедрения уведомлений (email/Telegram) для студентов о статусе заявки и напоминаний об оплате.
