# Система бронирования мест в общежитии

## Основная часть — описание функционала

### Назначение

Система бронирования позволяет абитуриентам и студентам самостоятельно забронировать свободное место в комнате общежития через öffentlich сайт, а администратору — обработать заявку в панели управления.

### Пользовательский сценарий (публичная часть)

1. **Выбор этажа.** Пользователь переходит на страницу «Планировка этажей» и выбирает интересующий этаж (со 2-го по 5-й). На экране отображается SVG-план этажа с реальными контурами помещений.

2. **Просмотр комнаты.** При нажатии на жилую комнату на плане открывается карточка комнаты, где показаны все места:
   - Зелёная иконка — место свободно, доступна кнопка «Забронировать»
   - Красная иконка — место занято

3. **Заполнение заявки.** При нажатии «Забронировать» открывается форма, в которой нужно указать:
   - ФИО (обязательно)
   - Курс (1–4)
   - Группу (обязательно, например «ИС-21»)
   - Телефон (необязательно)

   Система проверяет, что выбранное место всё ещё свободно и не забронировано другим пользователем.

4. **Подтверждение.** После успешной отправки пользователь видит сообщение: «Заявка отправлена! Администратор рассмотрит вашу заявку и свяжется с вами.» На этом публичная часть завершена.

### Административный сценарий (панель управления)

1. **Просмотр заявок.** Администратор заходит в раздел «Бронирования» панели управления. По умолчанию отображаются заявки со статусом «Ожидает». Доступны фильтры по имени и статусу, а также экспорт в Excel.

2. **Подтверждение брони.** Администратор нажимает «Подтвердить» — система автоматически создаёт запись студента (с теми же данными: ФИО, курс, группа, комната, место) и меняет статус брони на «Подтверждена». Студент считается заселённым.

3. **Отклонение брони.** Администратор нажимает «Отклонить» — статус брони меняется на «Отклонена», место снова становится доступным для бронирования.

4. **Мониторинг.** На главной странице (Dashboard) панели управления отображается счётчик новых бронирований. Если есть необработанные заявки, карточка подсвечивается.

### Защита от конфликтов

- Одно место не может быть забронировано дважды — система проверяет это как при создании заявки, так и при подтверждении.
- Подтвердить можно только заявку в статусе «Ожидает».
- Забронировать можно только жилую комнату (не санузел, не техническое помещение).

## Специальная часть — техническая реализация

### Модель данных

Бронирование хранится в таблице `Booking` (СУБД PostgreSQL, ORM Prisma):

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Int (PK) | Уникальный идентификатор |
| `fullName` | String | ФИО студента |
| `course` | Int | Курс (1–4) |
| `group` | String | Учебная группа |
| `phone` | String? | Телефон (опционально) |
| `roomId` | Int (FK) | Ссылка на комнату |
| `bedNumber` | Int | Номер места в комнате |
| `status` | BookingStatus | PENDING / CONFIRMED / REJECTED |
| `createdAt` | DateTime | Дата создания заявки |

Перечисление `BookingStatus`:
- `PENDING` — заявка ожидает рассмотрения администратором
- `CONFIRMED` — заявка одобрена, создана запись студента
- `REJECTED` — заявка отклонена

Связь: `Booking.roomId → Room.id` (одна комната может иметь несколько бронирований).

### API-маршруты

Серверная часть реализована на Express.js. Все маршруты смонтированы на `/api/bookings`.

#### `POST /api/bookings` — создание заявки (публичный, без авторизации)

**Входные данные** (JSON):
```json
{
  "fullName": "Иванов Иван",
  "course": 2,
  "group": "ИС-21",
  "phone": "+77771234567",
  "roomId": 42,
  "bedNumber": 3
}
```

**Алгоритм проверки:**
1. Все обязательные поля заполнены
2. Комната существует и имеет тип `RESIDENTIAL`
3. Номер места в диапазоне `[1, room.totalBeds]`
4. Место не занято активным студентом (`movedOut: null`)
5. На это место нет другой заявки со статусом `PENDING`

При нарушении любого условия возвращается `400` или `409` с описанием ошибки. При успехе — `201` с объектом бронирования, включая номер и этаж комнаты.

#### `GET /api/bookings` — список заявок (только для администратора)

Поддерживает query-параметры: `status`, `floor`, `search` (поиск по ФИО, регистронезависимый). Возвращает заявки с информацией о комнате, отсортированные по дате создания (новые сверху).

#### `POST /api/bookings/:id/confirm` — подтверждение (только для администратора)

Выполняется внутри транзакции Prisma (`$transaction`):

1. Повторно проверяется доступность места (защита от гонки условий — другой администратор мог подтвердить другую заявку на то же место)
2. Создаётся запись `Student` с копированием данных из заявки
3. Статус заявки меняется на `CONFIRMED`

Если место занято на момент подтверждения — возвращается `409`.

#### `POST /api/bookings/:id/reject` — отклонение (только для администратора)

Меняет статус заявки на `REJECTED`. Отклонить можно только заявку в статусе `PENDING`.

### Публичный фронтенд (React)

Реализован как конечный автомат состояний в компоненте `FloorsPage` (`App.jsx`):

```
FloorPlan → RoomView → BookingForm → BookingSuccess
```

**Управление состояниями:**
- `selectedRoom` — выбранная комната (объект с beds)
- `bookingBed` — номер места, которое бронируют (активирует BookingForm)
- `submittedBooking` — ответ сервера после успешного бронирования (активирует BookingSuccess)

Каждый переход сбрасывает последующие состояния. После успешного бронирования вызывается `fetchRoomMap` для обновления данных о доступности мест.

**Компоненты:**
- `FloorOne–Five.jsx` — SVG-планы этажей, через проп `onRoomClick(roomNumber)` передают номер комнаты
- `RoomView.jsx` — отрисовывает сетку мест с кнопками «Забронировать» на свободных
- `BookingForm.jsx` — форма с полями ФИО, курс, группа, телефон; валидация на клиенте и сервере
- `BookingSuccess.jsx` — сообщение об успехе с SVG-иконкой галочки

### Панель управления (CMS)

Страница `BookingsPage.jsx` — табличное представление со следующим функционалом:
- **Фильтры:** поиск по имени (текстовое поле), фильтр по статусу (выпадающий список), по умолчанию выбран `PENDING`
- **Таблица:** ФИО, Группа, Курс, Телефон, Комната (с этажом), Место, Дата создания, Статус (цветной бейдж), Действия
- **Кнопки действий** (для заявок PENDING): «Подтвердить» (зелёная), «Отклонить» (красная) — обе с подтверждением через `window.confirm`
- **Экспорт в Excel** через утилиту `exportToExcel`
- Дизайн унифицирован с страницами `/students` и `/payments` — белая шапка с границей, белая полоса фильтров, таблица с тенью и скруглением, hover-эффект строк, цветовые бейджи статусов

На главной странице (`DashboardPage`) отображается карточка «Новых бронирований» с количеством заявок в статусе PENDING. При наличии необработанных заявок карточка подсвечивается индиговой рамкой.
