## 1. Общая архитектура системы

Трёхуровневая архитектура:

```
[Публичный фронтенд]     [Панель управления (CMS)]     [База данных]
   React + Vite             React + Vite              PostgreSQL
   Порт 5173                Порт 5174                Порт 5432
        │                        │                        │
        └────────┬───────────────┘                        │
                 │                                        │
          [Backend API]                                   │
         Express.js :3001                                 │
         JWT-авторизация                                  │
         Prisma ORM ──────────────────────────────────────┘
```

- Публичный фронтенд: просмотр планировки этажей, бронирование мест (без авторизации)
- CMS: управление студентами, комнатами, оплатой, бронированиями (JWT-вход админа)
- Backend: 8 модулей маршрутов, Prisma ORM к PostgreSQL
- Авторизация: middleware `requireAuth` проверяет JWT-токен на всех админских маршрутах. Публичные маршруты: `POST /api/bookings` (создать заявку) и `POST /api/auth/login`

## 2. Публичный фронтенд — пользовательский поток

Два маршрута React Router:
- `/` — HomePage (главная страница с информацией)
- `/floors` — FloorsPage (интерактивный план этажей)

### Поток FloorsPage (конечный автомат):

```
┌─────────────┐     клик по комнате     ┌─────────────┐
│  FloorPlan  │ ───────────────────────→ │  RoomView   │
│  (SVG план) │ ←──── «Назад» ───────── │  (сетка мест)│
└─────────────┘                          └──────┬──────┘
                                                │
                                         клик «Забронировать»
                                                │
                                                ▼
                                         ┌─────────────┐
                                         │ BookingForm │
                                         │ (форма ФИО, │
                                         │ курс, группа)│
                                         └──────┬──────┘
                                                │
                                         успешный POST
                                                │
                                                ▼
                                         ┌──────────────┐
                                         │BookingSuccess│
                                         │«Заявка       │
                                         │отправлена!»  │
                                         └──────────────┘
```

### FloorPlan (SVG):
- 5 этажей, отдельные компоненты: FloorOne–FloorFive.jsx
- Каждая комната — path/rect с onClick → onRoomClick(roomNumber)
- Комнаты раскрашены по статусу (свободные/занятые/нежилые)
- Легенда с цветовыми обозначениями (сворачиваемая)
- FloorSelector — переключатель этажей (кнопки 2–5)

### RoomView:
- Карточка комнаты: номер, свободно/всего мест
- Сетка мест: BedIcon (зелёный — свободно, красный — занято)
- Кнопка «Забронировать» только на свободных местах
- Кнопка «Назад» возвращает к плану этажа

### BookingForm:
- Поля: ФИО, Курс (select 1-4), Группа, Телефон (опционально)
- POST /api/bookings → проверка доступности места
- Ошибки показываются инлайн, успех → BookingSuccess
- Кнопка «Отмена» возвращает к RoomView

### BookingSuccess:
- SVG-иконка галочки, «Заявка отправлена!»
- Инфо: комната, место
- «Администратор рассмотрит вашу заявку»
- Кнопка «К планировке» → сброс к FloorPlan

## 3. Панель управления (CMS) — структура и потоки

### Навигация (Layout — боковая панель):
- Главная (Dashboard)
- Студенты (/students)
- Бронирования (/bookings)
- Комнаты (/rooms)
- Оплата (/payments)
- Кнопка «Выйти»

### Dashboard (главная):
5 карточек-счётчиков (кликабельные, ведут на соответствующие разделы):
- Заселено студентов (всего / активных / ожидают подписания)
- Свободных мест (из N всего)
- Оплатили в [месяц] (N не оплатили)
- Ожидают подписания (заявления)
- Новых бронирований (подсветка рамкой если >0)

Плюс: лента недавней активности (activity feed — события: бронь, заселение, оплата), быстрые действия, шкала заполненности общежития.

### Студенты (/students):
```
┌─────────────────────────────────────────────┐
│ Заголовок: Студенты  [Excel] [Импорт] [+Добавить] │
├─────────────────────────────────────────────┤
│ Фильтры: поиск по имени | этаж (select)      │
├─────────────────────────────────────────────┤
│ Таблица: ФИО|Группа|Курс|Тел|Комната|Заселение|Статус|Действия │
│   ▲ клик по строке → раскрывается детальная карточка            │
│ Действия: Изменить | Печать | Подтвердить | Выселить            │
└─────────────────────────────────────────────┘
```
- Статусы: PENDING (ожидает подписания) / ACTIVE (заселён) / без комнаты (не заселён)
- «Печать» → react-to-print рендерит ApplicationPrint (шаблон заявления)
- «Подтвердить» → переводит PENDING → ACTIVE (документ подписан)
- Модальное окно StudentModal для создания/редактирования
- Импорт из Excel через ImportButton
- Кнопка «Печать списка дежурного» → DutyRosterPrint (таблица с чекбоксами)

### Бронирования (/bookings):
```
┌──────────────────────────────────────────────┐
│ Заголовок: Бронирования  [Excel]              │
├──────────────────────────────────────────────┤
│ Фильтры: поиск по имени | статус (по умолчанию PENDING) │
├──────────────────────────────────────────────┤
│ Таблица: ФИО|Группа|Курс|Тел|Комната|Место|Дата|Статус|Действия │
│ Статусы: PENDING(жёлтый) / CONFIRMED(зелёный) / REJECTED(красный) │
│ Действия (PENDING): [Подтвердить] [Отклонить]                     │
└──────────────────────────────────────────────┘
```
- Подтвердить → транзакция: создаётся Student, статус → CONFIRMED
- Отклонить → статус → REJECTED

### Комнаты (/rooms):
- Селектор этажа (кнопки 2|3|4|5, активная подсвечена indigo)
- Сетка карточек комнат: номер, тип (бейдж: жилая/нежилая/санузел/тех), занято/всего мест, точки-индикаторы мест
- Клик по комнате → детальный вид (двухколоночный): инфо о комнате + сетка мест с именами студентов

### Оплата (/payments):
- Фильтры: месяц | год | этаж
- Счётчики: N оплатили / M не оплатили
- Таблица: ФИО|Группа|Комната|Сумма (10 000 ₸)|Статус|Дата оплаты|Действия
- Действия: «Отметить оплату» / «Отменить»
- Платёж уникален по (studentId, month, year)

## 4. API — структура маршрутов

```
/api/auth      → login (POST), me (GET)           — JWT-аутентификация
/api/students  → CRUD + checkout (POST) + confirm (POST) — админ
/api/rooms     → list (GET) + detail/:id (GET)     — админ + публично для карты
/api/bookings  → create (POST — публичный), list/confirm/reject (админ)
/api/payments  → list (GET), mark (POST), cancel (DELETE) — админ
/api/stats     → dashboard counters (GET)          — админ
/api/activity  → recent events feed (GET)          — админ
/api/import    → batch student import (POST)       — админ
```

Авторизация: все маршруты кроме `POST /bookings` и `POST /auth/login` проходят через `requireAuth` middleware (проверка JWT).

## 5. База данных — связи

```
Admin ─────────── отдельная таблица (логин/пароль/имя), не связана с остальными

Room 1───* Student     (комната → студенты)
Room 1───* Booking     (комната → бронирования)
Student 1───* Payment  (студент → платежи)

Enums:
  RoomType:      RESIDENTIAL | NON_RESIDENTIAL | TOILET | TECH
  StudentStatus: PENDING | ACTIVE
  BookingStatus: PENDING | CONFIRMED | REJECTED
```

Ключевые ограничения:
- Payment: @@unique([studentId, month, year])
- Booking.bedNumber + RoomId + status=PENDING — уникальность проверяется в коде (не на уровне БД)
- Student.bedNumber + RoomId + movedOut=null — уникальность проверяется в коде
- room.type должен быть RESIDENTIAL для создания брони

## 6. Основные бизнес-процессы

### Заселение студента (полный цикл):
```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ Бронь от │───→│ Админ    │───→│ Печать   │───→│ Подпись  │───→ Студент
│ студента │    │подтвержд.│    │заявления │    │студентом │    ACTIVE
│ PENDING  │    │→ Student │    │(react-to │    │+ админ   │
│          │    │  PENDING │    │ -print)  │    │confirm   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
```
Альтернативный путь (без брони): админ создаёт студента вручную через StudentModal.

### Оплата (ежемесячный цикл):
- Фиксированная сумма: 10 000 ₸/мес
- Админ выбирает месяц/год → отмечает оплативших
- Уникальность: один студент не может оплатить один месяц дважды
- Статус оплаты виден в карточке студента (detail row) и на странице оплаты

### Выселение:
- Админ нажимает «Выселить» → movedOut = now, roomId = null, bedNumber = null
- Студент остаётся в базе (история), но не показывается в активных списках
- Место освобождается
